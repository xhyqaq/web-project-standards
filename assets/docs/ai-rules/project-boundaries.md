# Project Boundaries Rules

本文件规定项目边界，不规定具体技术方案。AI 不应替项目擅自决定认证、权限、部署、安全、状态管理、UI 框架等方案。

## 基本原则

- 先遵循项目已有做法，再考虑新增方案。
- 找不到现有做法时，先选择最小实现；高风险决策必须询问维护者。
- 不要因为当前任务方便，就绕过已有架构、权限、安全、日志或错误处理边界。
- 新增依赖、全局约定、目录结构、跨模块抽象前，必须确认项目已有模式不能解决问题。

## 必须先查的项目决策

涉及以下内容时，先查现有代码、配置、文档和调用链：

- 认证方式：登录态、session、token、cookie。
- 权限方式：菜单权限、按钮权限、接口权限、数据权限。
- 环境配置：`.env`、部署变量、public/private env。
- 请求方式：fetch wrapper、错误处理、重试、缓存。
- 表单方案：schema、校验、错误展示。
- 状态管理：URL state、server state、client state。
- UI 体系：组件库、token、图标、布局。
- 日志和监控：日志格式、request id、错误上报。
- 数据访问：repository、ORM、transaction、migration。

## 不能擅自决定的事

- 不擅自更换认证、权限、状态管理、请求库、UI 组件库、ORM、测试框架。
- 不擅自引入全局单例、全局状态、全局事件总线。
- 不擅自改变 API contract、数据库结构、错误码、权限语义。
- 不擅自把一次性页面逻辑抽成跨项目抽象。

## 安全底线

- 密钥、token、连接串、私有证书不能进入源码、前端 bundle、日志、错误响应、测试快照。
- 前端权限判断只能改善体验，不能作为最终安全边界。
- 后端受保护资源必须做最终权限校验。
- 外部输入必须校验后再进入业务逻辑。
- 错误响应不能暴露堆栈、SQL、内部服务地址或第三方原始错误。
- 日志必须脱敏。

## 找不到现有方案时

- 先复用本规范中的默认技术栈。
- 实现最小可行方案，不提前设计复杂抽象。
- 把假设写清楚。
- 涉及安全、权限、数据迁移、生产配置、外部服务时，先问维护者。

## 完成前检查

- 是否遵循了项目已有方案？
- 是否新增了不必要的依赖或抽象？
- 是否破坏了权限、安全、日志、错误处理边界？
- 是否把服务端私密信息暴露到了前端？
- 是否需要补测试或文档？
